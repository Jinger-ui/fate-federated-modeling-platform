# 论文版系统设计章节

题目：基于 FATE 的多方数据安全联合建模平台设计与实现

副标题：以银行与运营商信用风险预测为例

## 第4章 系统分析与总体设计

### 4.1 系统建设背景

随着金融科技和数字经济的发展，银行、运营商、医疗、政务、电商等行业积累了大量高价值数据。这些数据在提升信用评估、风险识别、精准营销和联合治理能力方面具有重要意义。然而，由于行业之间存在数据安全、隐私保护、合规监管以及商业竞争等多重限制，各机构难以直接共享原始数据，形成了典型的数据孤岛问题。

在信用风险预测场景中，银行通常掌握用户基础信息、信用历史、贷款记录、还款行为以及是否逾期等标签数据，运营商则掌握用户通信频次、套餐消费水平、在网时长、欠费次数等补充特征。若仅依赖银行单方数据，模型虽然能够完成基本的风险预测，但对用户行为模式的刻画仍然不够完整。若能在不暴露原始数据的前提下融合运营商行为特征，理论上可以进一步提升信用风险预测模型的准确性与泛化能力。

联邦学习提供了一种“数据可用不可见”的技术路线。该技术允许多方在各自本地保留原始数据，仅通过加密参数交换、中间结果协同或安全求交等方式完成联合建模。FATE 作为国内较具代表性的开源联邦学习框架，提供了纵向联邦学习、隐私集合求交、联邦逻辑回归、联邦 SecureBoost 和任务编排能力，适合作为本项目的联邦计算底座。

因此，本文拟设计并实现一套基于 FATE 的多方数据安全联合建模平台，通过模拟银行与运营商的纵向联邦学习场景，构建集样本对齐、联合建模、训练监控、结果评估和审计追踪于一体的平台系统。

### 4.2 系统需求分析

#### 4.2.1 功能需求分析

结合项目目标，系统需满足以下核心功能需求：

1. 支持平台用户登录、身份认证和角色权限控制。
2. 支持参与机构信息维护，包括银行方、运营商方及其联邦节点配置。
3. 支持数据资产登记，记录数据集名称、字段说明、主键字段、标签字段和版本信息。
4. 支持样本对齐任务创建与执行，完成跨机构交集样本识别。
5. 支持联邦建模任务配置与管理，提供 Hetero Logistic Regression 与 Hetero SecureBoost 两类算法选择。
6. 支持底层 FATE 流程调用，并对任务状态、日志和外部任务编号进行统一管理。
7. 支持训练过程监控和状态可视化展示。
8. 支持模型评估报告生成，展示 Accuracy、Precision、Recall、F1-score、AUC、KS、Loss 等指标。
9. 支持关键操作与敏感行为的审计留痕。
10. 支持首页看板展示平台运行情况和实验对比结果。

#### 4.2.2 非功能需求分析

除功能需求外，平台还需要满足以下非功能需求：

1. 安全性：系统必须确保原始数据不出域，接口访问受控，敏感操作留痕。
2. 可扩展性：支持新增机构、新增算法和新增实验任务类型。
3. 可维护性：采用模块化设计，便于后续迭代和代码维护。
4. 可观测性：对联邦任务执行状态、失败原因和评估结果具备良好可追踪能力。
5. 易用性：通过统一 Web 平台降低 FATE 使用门槛，便于教学演示和毕业设计答辩。

### 4.3 系统建设目标

本系统的建设目标可从业务目标、技术目标和工程目标三个方面展开。

在业务目标上，系统需要模拟银行与运营商的联合建模场景，在保护各方数据安全的前提下完成信用风险预测，并通过实验验证联邦联合建模优于单方建模的效果。

在技术目标上，系统需要以 FATE 为联邦计算引擎，以 Spring Boot 为平台业务中台，以 Vue 3 为前端交互层，实现联邦任务配置、执行、监控和结果展示的完整闭环。

在工程目标上，系统需要具备企业级后台管理系统的基本形态，包括统一认证、权限隔离、审计日志、标准 REST API、可部署架构和可复现实验方案，从而体现平台化和工程化能力。

### 4.4 系统可行性分析

#### 4.4.1 技术可行性

本项目选用的 Java 17、Spring Boot 3、Vue 3、MySQL、Docker 等技术均较为成熟，开发资料丰富，能够满足后台管理平台快速开发要求。FATE 提供了较完善的纵向联邦学习能力与 Pipeline 编排接口，可以支撑银行与运营商的联合建模实验。因此，从技术栈成熟度和可集成性角度看，项目具备良好的技术可行性。

#### 4.4.2 经济可行性

本项目基于开源技术栈实现，部署环境可采用单机 Docker Compose 方式搭建，硬件资源需求适中，整体成本较低。对于毕业设计和课程设计场景而言，无需采购商业化联邦平台即可完成设计与验证，具备良好的经济可行性。

#### 4.4.3 实施可行性

为降低一次性接入 FATE 的复杂度，项目采用三阶段实施路线：第一阶段使用模拟数据和本地脚本实现平台功能闭环，第二阶段接入 FATE Standalone 和 FATE Pipeline，第三阶段通过 FATE Flow API 实现任务自动化提交和结果回写。该路线实施风险较低，适合毕业设计周期安排。

### 4.5 系统总体设计思路

系统设计遵循“前后端分离、分层解耦、原始数据不出域、任务流程可追踪”的原则，整体上划分为展示层、业务层、联邦计算适配层、多方数据节点层和基础设施层。

展示层基于 Vue 3 实现页面展示与用户交互，负责完成任务创建、状态查看和图表展示。业务层基于 Spring Boot 构建，负责用户权限、机构管理、数据资产管理、任务编排、结果持久化和日志审计。联邦计算适配层负责将平台侧的业务任务转换为 FATE 可执行流程，并向上提供统一的提交、查询和解析接口。多方数据节点层代表银行与运营商两类参与方，实际业务数据保留在本地，不直接进入平台数据库。基础设施层提供 MySQL、Docker、FATE Standalone 以及可选的 Nginx 反向代理能力。

### 4.6 系统总体架构

系统总体架构如文字描述所示：

1. 用户通过浏览器访问平台前端页面。
2. 前端通过 REST API 调用 Spring Boot 后端服务。
3. 后端根据任务配置调用联邦计算适配层。
4. 联邦计算适配层选择本地模拟器、FATE Pipeline 或 FATE Flow API 执行任务。
5. 银行节点和运营商节点分别在本地参与 PSI 和联合建模，原始数据不出域。
6. 联邦训练结果、任务状态和评估指标回写至 MySQL，供平台前端展示。

该架构使系统既能体现联邦学习“隐私保护”的核心思想，也能体现平台工程化的控制与管理能力。

### 4.7 系统业务流程设计

本系统的典型业务流程可概括为以下步骤：

1. 管理员完成机构信息和平台用户初始化。
2. 各机构用户登记本地数据资产元信息。
3. 平台用户发起样本对齐任务，完成跨机构样本交集识别。
4. 平台用户在选择算法和参数后创建联邦建模任务。
5. 后端将任务配置转换为底层 FATE 执行请求。
6. FATE 在各参与方本地完成训练与评估。
7. 平台周期性同步任务状态和训练指标。
8. 前端展示模型评估报告和平台运行看板。

该流程覆盖了联邦建模从“数据准备”到“任务执行”再到“结果分析”的完整生命周期。

## 第5章 系统详细设计

### 5.1 用户登录与权限模块设计

用户登录与权限模块是平台安全控制的基础。系统采用 Spring Security + JWT 的组合方式实现无状态认证。用户登录时提交用户名和密码，后端完成身份认证后签发 Token。前端在后续请求中携带 Token，后端基于角色和权限编码完成接口访问控制。

角色建议至少包含平台管理员、机构操作员和审计员三类。平台管理员可进行全局管理，机构操作员主要负责本机构数据资产与联邦任务操作，审计员以只读方式查看运行与日志信息。该设计既符合后台系统常见 RBAC 权限模型，也便于体现平台的安全治理能力。

### 5.2 机构管理模块设计

机构管理模块用于维护参与联邦建模的组织实体，包括机构名称、机构编码、机构类型、partyId、负责人和节点配置。该模块的关键作用在于建立平台业务组织与 FATE 联邦角色之间的映射关系。

在本项目场景中，银行通常作为 guest 方持有标签，运营商通常作为 host 方提供补充特征。通过为机构绑定 partyId、节点地址和数据命名空间等信息，系统可以在创建 PSI 任务和联邦训练任务时自动生成参与方配置，降低人工配置出错概率。

### 5.3 数据资产管理模块设计

数据资产管理模块负责对参与联邦建模的数据集进行元数据管理，而非直接存储业务明文数据。系统保存的数据内容主要包括数据集名称、所属机构、数据源引用、主键字段、标签字段、字段说明、数据量、版本号和校验状态。

该设计有两个明显优势。其一，平台能够感知数据资产结构并为后续任务提供输入来源。其二，平台不直接接收用户原始业务记录，能够更好地满足“原始数据不出域”的设计原则。该模块是联邦计算场景下区别于传统机器学习平台的重要部分。

### 5.4 样本对齐模块设计

由于银行和运营商拥有的是不同特征维度的用户数据，训练前必须先识别双方共有样本。本项目采用 PSI 机制实现隐私集合求交，从而完成样本对齐。

用户在平台侧发起样本对齐任务时，需要选择双方数据资产、样本主键字段和任务名称。系统首先校验主键字段的类型和可用性，然后生成 PSI 任务配置。若处于第一阶段，平台调用本地脚本模拟交集结果；若处于第二阶段或第三阶段，则调用 FATE 完成真实 PSI 计算。最终系统只保存交集样本量、结果引用和任务状态，而不保存交集样本明文 ID。

该模块体现了联邦学习场景中的关键前置环节，同时也体现了平台对隐私计算过程的任务化管理能力。

### 5.5 联邦建模任务模块设计

联邦建模任务模块是平台的核心模块，负责对 Hetero Logistic Regression 和 Hetero SecureBoost 等联邦模型训练任务进行统一管理。用户需要填写任务名称、算法类型、参与机构、关联 PSI 任务、数据资产、特征配置和模型参数。系统会对参数完整性、数据依赖关系和角色配置进行校验，校验通过后生成平台任务记录。

任务创建完成后，平台支持对任务执行提交、重试和状态查看。训练任务状态建议划分为待创建、待提交、运行中、成功、失败和已取消等，以便前端更直观地展示任务进展。该模块通过任务实体将平台业务逻辑与底层 FATE 执行逻辑隔离开，便于后续扩展更多算法或实验类型。

### 5.6 FATE 调用模块设计

FATE 调用模块位于业务层与联邦计算层之间，是平台工程化集成的重要体现。该模块的主要职责包括读取联邦任务配置、生成 Pipeline 脚本或 Flow API 请求、提交训练任务、保存外部任务编号、查询执行状态、解析训练指标与日志，并将结果标准化后回写数据库。

在实现上，为降低耦合度，建议将该模块封装为独立的 FATE Adapter Service。业务模块只需调用统一接口，而不直接处理脚本生成和命令拼接。该做法更符合企业项目的分层设计思想，也有利于后续替换底层联邦框架或扩展运行模式。

### 5.7 训练状态监控模块设计

训练状态监控模块面向用户展示联邦任务的运行过程，包括任务当前状态、外部 jobId、开始时间、结束时间、耗时、阶段日志和异常信息。该模块通常通过定时调度方式周期性查询 FATE Flow 执行结果，并将其转换为平台统一的状态值，例如 RUNNING、SUCCESS、FAILED。

前端页面可以使用状态卡片、时间轴和日志窗口来展示训练过程，从而增强系统的可观测性。该模块一方面有助于提升平台可用性，另一方面也有助于毕业设计答辩时直观展示系统“不是黑盒执行”的工程特征。

### 5.8 模型评估报告模块设计

模型评估报告模块负责对训练完成后的模型结果进行展示与比较。平台需要从 FATE 输出中提取 Accuracy、Precision、Recall、F1-score、AUC、KS、Loss 等指标，并将其标准化保存到数据库中。同时，对于 ROC 曲线、Loss 曲线等图形化结果，可采用 JSON 结构存储曲线点数据，供前端 ECharts 直接渲染。

评估报告模块不仅服务于系统可视化展示，也是论文实验分析的重要数据来源。通过该模块，用户可以将银行单方训练、运营商单方训练和银行加运营商联邦训练三组实验结果放在同一界面中进行对比分析，更清晰地说明联邦建模带来的效果提升。

### 5.9 审计日志模块设计

审计日志模块用于记录平台中的关键行为，包括登录行为、任务提交行为、权限变更行为、异常访问行为以及训练结果同步行为等。实现方式上可结合 Spring AOP、过滤器和统一异常处理器记录日志。

在日志内容上，建议记录操作者、所属机构、操作模块、请求路径、请求方法、参数摘要、响应状态、耗时、IP 地址和操作时间等字段。需要特别注意的是，日志中不能记录明文密码和完整敏感数据。通过审计日志设计，平台能够体现系统安全治理、问题追踪和可追责能力。

### 5.10 首页看板模块设计

首页看板模块用于集中展示平台总体运行情况与实验结果摘要。建议页面展示内容包括机构数量、数据资产数量、任务总量、运行中任务数量、任务成功率、最近实验结果、模型效果对比图和近期操作记录等。

该模块既承担平台运营概览功能，也承担项目演示与答辩展示功能。特别是“单方模型与联邦模型 AUC 对比图”“样本对齐任务趋势图”“任务执行成功率”这类图表，能够直观体现项目的实际效果与平台价值。

### 5.11 数据库设计思想

数据库设计遵循“平台元数据入库、原始数据不入库、结构化表与扩展 JSON 相结合”的原则。用户、角色、权限、机构、数据资产、PSI 任务、联邦任务、模型评估报告和审计日志等内容需要持久化存储。训练参数、曲线数据和外部任务响应等可采用 JSON 字段增强扩展性。

该设计既能满足管理后台对结构化查询和统计分析的需求，也能应对联邦任务参数差异较大的场景。

### 5.12 接口设计思想

系统采用 REST 风格接口设计，前后端通过 JSON 格式数据交互。为了提高一致性，建议统一响应结构为 `code`、`message` 和 `data`。对于联邦任务和样本对齐任务这类长耗时操作，平台应采用“异步提交 + 状态轮询”的方式，避免请求阻塞。

此外，接口层需要结合用户身份与机构归属进行数据隔离，确保机构操作员只能访问本机构相关数据。该设计可体现系统在多租户和数据边界方面的考虑。

### 5.13 安全设计思想

系统安全设计从身份认证、访问控制、数据保护、日志审计和部署隔离五个层面展开。身份认证采用 Spring Security + JWT，访问控制采用 RBAC 模型，数据保护强调原始数据不出域和敏感字段脱敏，日志审计用于记录关键操作轨迹，部署隔离则通过 Docker 容器与环境变量管理降低配置暴露风险。

在联邦学习系统中，安全设计不只是平台账号安全，还包括对数据边界的严格控制。因此，平台层需要始终坚持“只管理任务和结果，不管理参与方原始业务明文数据”的设计底线。

## 第6章 系统实现与实验方案

### 6.1 分阶段实现路线

为兼顾项目周期与实现可行性，系统采用分阶段实现策略。

第一阶段，使用模拟数据和本地脚本模拟联合训练流程，重点验证平台页面、接口、任务状态和结果展示闭环。该阶段有助于快速搭建可演示版本。

第二阶段，接入 FATE Standalone 环境，并通过 FATE Pipeline 编排 Reader、DataTransform、Intersection、HeteroLR 或 Hetero SecureBoost、Evaluation 等组件，完成真实联邦训练。

第三阶段，通过 FATE Flow API 或命令行方式提交任务、查询状态、解析日志和训练指标，并回写 MySQL，实现联邦训练能力的平台化集成。

### 6.2 实验设计

为验证联合建模价值，本文设计三组实验。

第一组实验为银行单方训练，仅使用银行侧特征和标签构建模型，作为基线实验。

第二组实验为运营商单方训练，使用运营商侧特征进行实验模拟，重点分析行为特征的独立建模能力。

第三组实验为银行与运营商联邦联合训练，通过 PSI 获取交集样本，并在纵向联邦场景下完成联合建模。通过对三组实验的指标对比，可以验证联邦学习在不共享原始数据的前提下是否提升模型性能。

### 6.3 评价指标

实验主要采用以下指标进行评价：

1. Accuracy，用于衡量整体预测准确率。
2. Precision，用于衡量预测为高风险样本中的实际高风险比例。
3. Recall，用于衡量高风险样本被识别出来的比例。
4. F1-score，用于综合评价 Precision 与 Recall。
5. AUC，用于衡量模型区分正负样本的能力。
6. KS，用于衡量风控模型的区分度。
7. Loss，用于观察模型训练收敛情况。
